# 상세개발계획: 범용 DART 정보 검색기 (웹앱)

> 기준 문서: [PRD.md](PRD.md) — Phase 1 (데이터 수집기) 범위를 **웹앱**으로 구현한다.
> 작성일: 2026-07-14

---

## 1. 개요 및 목표

- 지역 / 매출액 범위 / 업종 조건을 **웹 화면에서 입력**하면, OpenDART API 기반으로
  외부감사대상 법인의 기본정보 + 요약 재무정보(당기·전기)를 수집하여
  **화면 테이블로 조회 + CSV/Excel 다운로드**까지 제공하는 로컬 웹앱.
- 수집 작업은 수 분~수 시간이 걸릴 수 있으므로 **백그라운드 작업(Job) 구조**로 설계하고,
  진행률을 화면에서 실시간 확인할 수 있게 한다.
- 단일 사용자 로컬 실행(`localhost`)을 기본 전제로 하되, 추후 사내 서버 배포가 가능한 구조로 만든다.

### Phase 1 완료 기준 (PRD 8장 → 웹앱 기준으로 재정의)

| # | 완료 기준 |
|---|---|
| 1 | 웹 화면에서 지역/매출액/업종 조건 입력 및 검색 실행 가능 |
| 2 | 조건에 맞는 회사 목록 자동 추출 (백그라운드 Job) |
| 3 | 회사별 기본정보 7항목 수집 (업체명/주소/전화번호/대표자/업종명·코드/결산기준일/감사의견) |
| 4 | 회사별 당기·전기 요약 재무정보 13항목 파싱 (PRD 3-2절 전 항목) |
| 5 | 결과 화면 조회 + CSV/Excel 다운로드 |
| 6 | 진행률/오류 현황 표시 및 실패 건 재시도 기능 |
| 7 | 샘플 10개사 수동 검수로 파싱 정확도 확인 |

## 4. 데이터 수집 파이프라인 상세

PRD 5장의 8단계를 Job 실행 단계(STEP)로 매핑한다. 각 STEP 완료 시 DB에 체크포인트를
저장하여 **중단 후 이어하기(resume)** 가 가능해야 한다 (일일 호출한도 20,000건 대응).

| STEP | 내용 | 사용 API | 산출 |
|---|---|---|---|
| 0 | 조건 입력 검증, Job 생성 | - | `jobs` 레코드 |
| 1 | corpCode 캐시 확인/갱신 (7일 경과 시 재다운로드) | corpCode.xml | `corp_cache` 테이블 |
| 2 | 대상 기간 내 "외부감사관련(pblntf_ty=F)" 공시 목록 수집 (페이징 순회) | list.json | 후보 회사 목록 + 접수번호(rcept_no) |
| 3 | **지역 사전 추림**(금융위 주소 DB 매칭, 4-1절 대응 1) → 추려진 후보만 기업개황 조회로 확정 + `corp_profiles` 캐시 적재 | 금융위 기업기본정보, company.json | 필터 통과 회사 목록 |
| 4 | 감사보고서 원본 다운로드 (zip 해제 → XML/PDF 판별) | document.xml | 원문 파일 (로컬 캐시 디렉터리) |
| 5 | 재무제표 파싱 (당기·전기 13항목) + 감사의견 추출 | - | `results` 레코드 (파싱 상태 포함) |
| 6 | **매출액 범위 필터** 적용, 결과 확정 | - | 최종 결과셋 |
| 7 | **최근 N년 재무이력 수집**(`excluded_by_revenue=0`인 최종 결과만 대상, 2026-07-15 추가) | list.json(corp_code 지정) + document.xml | `financial_snapshots` 레코드 (회사×회계연도) |

> **STEP 7 구현 메모 (2026-07-15 추가 — §4-6 참고)**: 사용자가 "최근 N년치
> 재무 이력"을 요청해 STEP 6 뒤에 추가했다. STEP 3(FSC 사전 추림)와 같은
> "값비싼 호출은 후보를 최대한 추린 뒤에" 원칙을 재사용해, STEP 1~6을 다
> 통과한 최종 결과만 대상으로 한다. Job의 DONE 판정 시점이 STEP 6에서 STEP 7
> 완료 시점으로 다시 이동했다.

> **M2 구현 메모 (resume의 STEP별 의미 차이)**: §5 스키마에는 STEP 2(공시 목록) 조회
> 결과를 위한 전용 테이블이 없다. 그래서 STEP 1(corp_cache TTL 체크)과 STEP 2
> (list.json 페이징)는 resume 시 **항상 처음부터 다시 실행**한다 — 둘 다 멱등이고
> 호출 비용이 낮기 때문(STEP1은 TTL 이내면 즉시 skip, STEP2는 페이지 수만큼만 호출).
> 실제로 호출 비용이 큰 STEP 3(company.json, 후보 수만큼)과 STEP 4(document.xml,
> 필터 통과 건수만큼)만 각각 `corp_profiles` TTL 캐시와 `DOCUMENT_CACHE_DIR` 로컬
> 파일 캐시로 "이미 처리된 건은 재호출하지 않는" 방식의 진짜 resume을 구현한다.
> `results` 테이블도 이미 삽입된 corp_code는 재삽입하지 않는다
> (`app/core/pipeline.py`).

### 4-1. 지역 필터 — 구조적 제약과 전역 캐시 전략 ★

> **M2 구현 현황(2026-07-15 갱신)**: 대응 1(금융위 API)의 커버리지 스파이크가
> 완료되어(경남 표본 19/19 매칭, 100%) **대응 1을 채택·구현했다.**
> `app/core/pipeline.py`의 `_resolve_candidate_profile()`이 캐시 미스 시
> `FscCorpInfoClient.get_corp_basic_info()`로 회사명을 먼저 조회해 지역이
> 명백히 다르면 DART `company.json` 호출을 생략하도록 바뀌었다(아래 "결합
> 흐름(실제 구현)" 참고). STEP 3 호출부(`_run_region_industry_filter`)의
> 나머지 로직(지역/업종 매칭, results 선삽입)은 그대로 유지했다. 별도의
> 추상클래스나 플러그인 시스템은 과설계로 판단해 도입하지 않았다.
>
> **원래 계획(아래 "결합 흐름" 1~3단계, §8 M2 체크리스트)과의 차이**: 당초에는
> 금융위 API로 **전체 주소 DB를 사전 일괄 구축**(4만 개사 ≈ 400회 호출)한 뒤
> 회사명으로 매칭하는 방식을 상정했다. 실제로는 이를 별도 배치 스크립트로
> 만들지 않고, **Job 실행 중 STEP 3에서 후보 1건마다 캐시 미스 시점에
> 그때그때 FSC 회사명 조회 1건을 먼저 호출**하는 방식으로 단순화했다 — 결과는
> 동일(지역이 명백히 다른 후보의 DART 호출 생략)하지만 사전 일괄 적재
> 스크립트/전용 테이블이 없어도 되므로 구현이 더 단순하고, `corp_profiles`가
> Job을 거치며 점진적으로 채워지는 기존 대응 2의 방식과도 일관된다(과설계
> 방지). 새 DB 테이블은 추가하지 않았다 — `corp_profiles`에 FSC로만 확인된
> 부분 데이터(`sido`/`sigungu`만 채우고 `phone`/`ceo_name`/`induty_code`는
> `NULL`)를 upsert하는 것으로 충분했다.

**제약**: OpenDART API에는 지역 검색 파라미터가 없다. 회사의 주소는 기업개황
(company.json)을 **회사별 1건씩 조회**해야만 알 수 있다. 따라서 "김해만 검색"하더라도
어느 회사가 김해인지 판별하기 위해 후보 전체(대상 기간 내 감사보고서 제출 회사,
연간 약 3~4만 개사)의 기업개황이 필요하며, **최초 실행 비용은 김해 단독이든 경남
전체든 동일**하다 (일일 한도 20,000건 → 최초 1회 약 2일 소요).

**대응 1 (채택, 2026-07-15) — 금융위원회_기업기본정보 API로 지역 사전 추림**:
- 공공데이터포털(data.go.kr)의 **금융위원회_기업기본정보조회 API**(`getCorpOutline_V2`,
  무료, 별도 키 발급)는 DART 공시법인의 기본정보(본점 주소 `enpBsadr` 포함)를
  회사명(`corpNm`) 조회로 확인 가능하고, OpenDART와 **별도 쿼터**를 사용한다
  (`app/core/dart_client.py`의 `FscCorpInfoClient`, `DartClient`의 `api_usage`
  카운터와 무관).
- 원래 계획한 흐름(참고용, 실제 구현과는 아래 "실제 구현" 문단 참고):
  1. STEP 2 (DART list.json)로 후보 회사 목록 확보 — 기존과 동일
  2. 금융위 주소 DB와 **회사명 매칭** → 지역 후보만 추림 (3~4만 → 수백 개사)
     - 매칭 전 이름 정규화: "(주)/주식회사" 제거, 공백/괄호 통일 (`filters.normalize_corp_name`)
  3. 추려진 후보만 DART company.json 호출 → 주소 최종 확정(오탐 제거) +
     전화번호·대표자 수집 (Phase 1 필수 수집 항목이므로 낭비 호출 아님)
- **✅ 스파이크 실행 완료 (2026-07-15)**: 경상남도 소재 비상장 외감법인 표본 19건
  (목표 20건) 전수 매칭, **커버리지율 100%** → **대응 1 채택 확정**. 실행 스크립트는
  `backend/app/core/spike_financial_committee_coverage.py`. 응답 스키마(`response.body.
  items.item`이 리스트, 결과 없음일 때도 빈 리스트를 감싼 dict로 와서 truthy 판정만으로는
  매칭 여부를 판별할 수 없음)를 실측으로 확인해 스크립트의 판정 로직을 수정했다.
- **✅ 실제 구현 완료 (2026-07-15)**: `app/core/pipeline.py::_resolve_candidate_profile()`이
  캐시 미스마다 아래 3분기로 동작한다(별도의 전체 사전 일괄 적재 배치는 만들지
  않았다 — 과설계 방지, 위 "M2 구현 현황" 참고):
  1. `FscCorpInfoClient.get_corp_basic_info(corp_nm=정규화된 회사명)`으로 조회.
  2. 매칭되고 지역이 맞으면(또는 `cond_region`이 비어 지역 필터 자체가 없으면)
     기존과 동일하게 DART `company.json`을 호출해 전화번호/대표자/업종코드까지
     확정하고 `corp_profiles`를 풀 데이터로 upsert.
  3. 매칭됐는데 지역이 명백히 다르면 `company.json` 호출을 생략하고, FSC로
     확인한 `sido`/`sigungu`만 `corp_profiles`에 부분 upsert(나머지 컬럼은
     `NULL`) — `region_matches()`가 이 후보를 자연히 탈락시킨다.
  4. FSC에서 매칭되지 않거나(이름 검색 결과 없음) FSC 호출 자체가 실패(네트워크
     오류 등)하면 **보수적으로** 기존처럼 `company.json`을 직접 호출해 확정한다
     (놓치는 것보다 호출 1건 낭비가 낫다는 판단, 스파이크 커버리지 100%에도
     불구하고 안전망으로 유지).
  단위 테스트: `backend/tests/test_pipeline.py`의 `test_resolve_candidate_profile_fsc_*`
  4종(지역 일치/불일치/미매칭/FSC 실패) + `test_run_region_industry_filter_fsc_reduces_company_json_calls`
  + `test_run_job_fsc_prefilter_reduces_dart_company_calls_end_to_end`(run_job 전체 e2e).
- 검토 후 제외한 대안: DART 홈페이지 지역검색 크롤링(약관 위반, PRD 제외),
  중소기업현황정보시스템 sminfo(지역 검색 가능하나 자동화용 API 미제공),
  크레탑/KISLINE 등 상용 DB(확실하나 유료 — 필요 시 Phase 2에서 재검토).

**대응 2 (폴백) — `corp_profiles` 전역 캐시**:
- 기업개황 조회 결과(주소/시도/시군구/업종코드/대표자/전화)를 Job과 무관한
  **전역 마스터 테이블**에 영구 저장. 한 번 조회한 회사는 재조회하지 않는다.
- 이후 검색(김해 → 양산 → 창원 등 조건 변경 재검색)은 캐시만으로 즉시 지역 필터링
  → STEP 3의 API 호출이 신규 회사분만으로 급감.
- 주소·업종은 변동이 드물므로 갱신 주기는 길게(기본 180일, 설정 가능).
- 캐시 적재 시 주소에서 시도/시군구를 파싱해 별도 컬럼으로 저장 → 필터는 컬럼
  매칭으로 수행 (매번 문자열 검색하지 않음).
- "예상 규모 미리보기"에서 **캐시 보유율**(예: 후보 32,000개사 중 캐시 28,000건,
  신규 호출 필요 4,000건)을 함께 표시해 사용자가 소요 시간을 예측할 수 있게 한다.

**필터 로직**:
- 프론트에서 **광역시도 선택 → 시/군/구 다중 선택** 2단계 UI 제공.
- 주소 표기 편차 대응: "경상남도/경남" 등 약칭 매핑 테이블을 `filters.py`에 상수로 정의.

### 4-2. 업종 필터

- 기업개황 `induty_code`(표준산업분류) prefix 매칭.
- 프론트에서 대분류 → 중분류 트리 선택 UI (복수 선택). 분류코드 마스터는
  통계청 KSIC 10차 목록을 JSON으로 백엔드에 내장(`meta.py`에서 서빙).

### 4-3. 매출액 필터

- **사전 필터 불가**: 비상장 외감법인은 재무제표 API가 지원되지 않아(PRD 4장),
  원문을 파싱하기 전에는 매출액을 알 수 없다. 따라서 매출액은 구조적으로
  **마지막 단계의 사후 필터**일 수밖에 없다. 대신 지역·업종 필터(STEP 3)를 먼저
  통과시켜 다운로드·파싱 대상을 최소화한다.
- 파싱된 당기 매출액 기준 `min ≤ 매출액 ≤ max` (억원 단위 입력 → 원 단위 환산).
- 사후 필터이므로 필터 탈락 건도 `results`에 남기되 `excluded_by_revenue` 플래그 처리
  (재검색 시 재파싱 없이 재사용).

### 4-4. 재무제표 파싱 전략 (핵심 리스크 구간)

> **M3 구현 현황(2026-07-15) — 실측 메모**: 실제 DART API로 원문 30건을
> 확보해(2026년 4~6월 접수분 25건 + 2012년 초 접수분 5건,
> `backend/tests/fixtures/manifest.json`) 서식을 조사한 결과 **30건 전부
> XML**이었다 — 2012년 초 원문까지도 document.xml API가 XML로 반환했다.
> 계획 당시 우려했던 PDF/HWP 비중은 이 표본에서는 0%였다(이 저장소가 실제로
> 마주치지 못했을 뿐 존재 자체가 없다는 뜻은 아니므로, `pdf_parser.py`는
> best-effort로 구현해 두었고 M5에서 실물이 나오면 검증한다). 실측으로 확인된
> 원문 구조와 서식 편차(회사마다 로마숫자 표기가 유니코드/아스키로 갈리거나,
> 각주 참조가 "(주석13)"/"(주6)"로 축약 방식이 다르거나, 매출원가/판관비를
> 아예 구분하지 않는 "영업수익/영업비용" 서비스업 서식이 있는 등)는
> `app/parsers/base.py`/`xml_parser.py`/`audit_opinion.py` 모듈독스트링과
> `backend/tests/test_parsers.py`에 근거 라인과 함께 기록해 두었다.

1. **1순위 — XML 파싱**: document.xml이 반환하는 원본 zip에는 DART 접수 XML이
   포함되는 경우가 대부분. `<TABLE>` 구조에서 재무상태표/손익계산서 표를 탐색.
   실제 구현(`app/parsers/xml_parser.py`)은 `<TITLE>` 텍스트로 재무상태표/
   손익계산서 구간을 판별한 뒤 그 구간의 첫 `ACLASS="FINANCE"` 테이블만
   사용한다(자본변동표/현금흐름표도 같은 ACLASS를 쓰므로 혼입 방지). 원문
   본문에 이스케이프 안 된 특수문자가 섞여 엄격 XML 파싱이 실패하는 경우가
   실측 30건 중 다수 있어 `lxml.etree.XMLParser(recover=True)`로 손상된
   부분만 건너뛰고 나머지 구조를 살린다.
2. **2순위 — PDF 파싱**: 원본이 PDF인 경우 `pdfplumber`로 표 추출
   (`app/parsers/pdf_parser.py` — 위 실측 메모대로 실제 PDF 표본으로
   검증되지 않은 best-effort 구현).
3. **HWP**: Phase 1에서는 파싱 시도하되 실패 시 `PARSE_FAILED(HWP)` 상태로 기록만
   하고 넘어간다 (수동 검수 대상 목록에 포함). 미구현 상태 유지.
4. **계정과목 정규화 사전**: "매출액/영업수익/수익(매출액)", "당기순이익/당기순손실"
   등 표기 변형 → 표준 13항목으로 매핑하는 사전을 `parsers/base.py`에 정의하고,
   검수 과정에서 지속 보강한다. "영업손실"/"당기순손실"처럼 손실이 명시된
   행은 원문 숫자가 부호 없이 양수로 찍혀 있어 저장 시 부호를 뒤집는다.
   "매출총이익율(%)"은 원문의 "매출총이익"(금액) 행을 그대로 쓰지 않고
   매출액/매출원가로부터 계산한다(PRD 3-2절 정의, `compute_gross_margin()`).
5. 당기/전기 컬럼 판별: 헤더의 "제 N 기" 숫자 비교 또는 좌→우 순서 규칙 적용.
   실제 구현은 THEAD의 TH `COLSPAN`으로 당기/전기 열 개수를 판별하고, 각
   TR의 값 셀을 그 열 개수만큼 앞/뒤로 나눠 그룹 내 첫 번째 비어있지 않은
   셀을 채택한다(같은 그룹 안에 "상세/합계" 두 하위 열이 있는 서식이 실측
   대부분이었다).
6. 모든 파싱 결과에 `parse_status`(OK / PARTIAL / FAILED)와 `parse_note`를 남겨
   화면에서 수동 검수 대상을 바로 필터링할 수 있게 한다. 실측 기준: 재무제표
   자체가 첨부되지 않은 경우(의견거절 등, ACLASS="FINANCE" 테이블 없음)와
   서비스업 서식이라 매출원가/판관비 항목이 구조적으로 없는 경우 모두
   PARTIAL로 판정하고(파싱 실패가 아니라 원문에 없는 것), XML 구문이 복구
   불가할 정도로 손상된 경우만 FAILED로 판정한다.

### 4-5. API 호출량 관리

- `dart_client.py`에 일일 호출 카운터(DB 저장) 내장. 상한(기본 19,000, 여유분 확보)
  도달 시 Job을 `PAUSED_QUOTA` 상태로 자동 전환하고 다음 날 재개 버튼 제공.
- 요청 간 딜레이(기본 0.1초) 및 429/5xx 지수 백오프 재시도(최대 3회).

### 4-6. 최근 N년 재무이력 확장 (STEP 7, 2026-07-15 추가)

- **배경**: Phase 1 원래 스코프(PRD §3-2, §4-3)는 감사보고서 1건에서 나오는
  당기·전기(2개년)만 수집한다. 사용자가 이를 확장해 회사별 **최근 N년치**
  이력을 원해, 감사보고서가 당기·전기를 비교식으로 공시한다는 점(보고서
  1건 = 2개년)에 착안해 N을 **2/4/6/10**(짝수만) 옵션으로, 기본값 **4**로
  확정했다. `results` 테이블의 `_cur`/`_prv` 컬럼은 전혀 바꾸지 않는다
  ("가장 최근 감사보고서 1건의 당기·전기") — 새 테이블 `financial_snapshots`
  (§5)가 회사×회계연도 단위로 이력을 별도 보관한다.
- **"필터 통과 후에만" 원칙**: §4-1(대응 1, FSC 사전 추림)과 동일한 철학 —
  전체 후보가 아니라 STEP 1~6을 다 통과하고 `excluded_by_revenue=0`인
  최종 결과만 대상으로 하여, 쿼터 영향이 최종 결과 건수에만 비례하게 한다.
- **실측(2026-07-15): `list.json`에 `corp_code`를 지정하면 3개월 조회기간
  제한이 사라진다.** STEP 2가 겪은 "corp_code 없이는 90일 제한"(§4-1
  근처 실측 메모)과 달리, corp_code를 지정한 조회는 10년 범위도 한 번에
  성공했다(`corp_code=01552935` 표본, 2021~2025 회계연도 5건 응답). 따라서
  STEP 7은 `_split_period_into_windows()`를 사용하지 않고 단일 기간
  (`bgn_de`는 오늘로부터 `history_years//2+2`년 전 1/1)으로 조회한다.
- **처리 순서는 최신 공시 → 과거 공시 순(newest-first)**이다. 오래된
  것부터 훑으며 "목표 연도수만큼 모이면 중단"하면 오히려 가장 최근 연도를
  놓칠 수 있어(예: N=4년 목표인데 과거 보고서부터 채우다 4개 연도를 다
  채우면 정작 최신 보고서를 열어보지 않게 됨) newest-first로 설계했다.
  정정 공시가 있어도 "연도별로 이미 확보했으면 건너뜀" 규칙 덕에 최신
  rcept_no 값이 자연히 유지된다.
- **회계연도 판정**: 원문에는 당기 결산기준일(PERIODTO)만 있고 전기
  결산기준일은 별도 마커가 없다 — 당기 연도는 PERIODTO 연도, 전기 연도는
  "당기 연도 - 1"로 계산한다(연 1회 정기감사 가정).
- **다운로드/파싱은 STEP 4/5 로직을 100% 재사용**한다(`_ensure_document_cached()`로
  STEP 4/7이 원문 다운로드+로컬 캐시 로직을 공유하도록 추출, 새 파서는
  만들지 않음). resume은 회사(result)별 `financial_snapshots`의 distinct
  fiscal_year 수가 `history_years` 이상이면 그 회사는 list.json조차
  호출하지 않는 방식으로 구현했다.
- **Job 완료(DONE) 판정 시점**이 STEP 6에서 STEP 7 완료 시점으로 다시
  이동했다(M3에서 STEP 4 → STEP 6로 옮긴 전례와 동일한 패턴).
- 신규 API `GET /api/jobs/{id}/results/{result_id}/history`(§6)를 추가했다
  — 기존 `/results` 목록·`/export`는 무겁게 만들지 않기 위해 그대로 두고,
  상세 조회에서만 이력을 lazy-load하게 분리했다.

## 5. DB 스키마 (SQLite)

```sql
-- corpCode 캐시
CREATE TABLE corp_cache (
  corp_code   TEXT PRIMARY KEY,   -- 8자리 고유번호
  corp_name   TEXT NOT NULL,
  stock_code  TEXT,               -- 상장사만 존재 (비상장 필터에 활용)
  modify_date TEXT
);
CREATE TABLE cache_meta (key TEXT PRIMARY KEY, value TEXT);  -- corp_cache 갱신일 등

-- 기업개황 전역 캐시 (4-1절 — Job과 무관하게 영구 보존, 재검색 시 재사용)
CREATE TABLE corp_profiles (
  corp_code   TEXT PRIMARY KEY,
  corp_name   TEXT, address TEXT,
  sido        TEXT, sigungu TEXT,   -- 주소에서 파싱해 저장 → 지역 필터는 컬럼 매칭
  induty_code TEXT, phone TEXT, ceo_name TEXT,
  fetched_at  TEXT                  -- 갱신 주기 판단용 (기본 180일)
);

-- 검색 Job
CREATE TABLE jobs (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  created_at    TEXT, name TEXT,
  cond_region   TEXT,   -- JSON: {"sido":"경남","sigungu":["김해시","양산시"]}
  cond_revenue  TEXT,   -- JSON: {"min_krw":6000000000,"max_krw":15000000000}
  cond_industry TEXT,   -- JSON: ["C25","C29"]
  cond_period   TEXT,   -- JSON: {"bgn_de":"20250101","end_de":"20251231"}
  status        TEXT,   -- PENDING/RUNNING/PAUSED_QUOTA/DONE/FAILED/CANCELLED
  current_step  INTEGER, progress_done INTEGER, progress_total INTEGER,
  error_msg     TEXT,
  history_years INTEGER DEFAULT 4  -- STEP 7 목표 연도수(2/4/6/10만 허용, 2026-07-15 추가)
);

-- 수집 결과 (회사 1건 = 1행)
CREATE TABLE results (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  job_id      INTEGER REFERENCES jobs(id),
  corp_code   TEXT, rcept_no TEXT,
  -- 기본정보 (PRD 3-1)
  corp_name   TEXT, address TEXT, phone TEXT, ceo_name TEXT,
  induty_code TEXT, induty_name TEXT,
  fiscal_date TEXT,          -- 결산기준일
  audit_opinion TEXT,        -- 적정/한정/부적정/의견거절
  -- 요약 재무 (PRD 3-2) : 당기(_cur) / 전기(_prv), 단위: 원
  current_assets_cur      INTEGER, current_assets_prv      INTEGER,
  noncurrent_assets_cur   INTEGER, noncurrent_assets_prv   INTEGER,
  total_assets_cur        INTEGER, total_assets_prv        INTEGER,
  current_liab_cur        INTEGER, current_liab_prv        INTEGER,
  noncurrent_liab_cur     INTEGER, noncurrent_liab_prv     INTEGER,
  total_liab_cur          INTEGER, total_liab_prv          INTEGER,
  total_equity_cur        INTEGER, total_equity_prv        INTEGER,
  revenue_cur             INTEGER, revenue_prv             INTEGER,
  cogs_cur                INTEGER, cogs_prv                INTEGER,
  gross_margin_cur        REAL,    gross_margin_prv        REAL,   -- 매출총이익율(%)
  sga_cur                 INTEGER, sga_prv                 INTEGER,
  operating_income_cur    INTEGER, operating_income_prv    INTEGER,
  net_income_cur          INTEGER, net_income_prv          INTEGER,
  -- 상태
  parse_status TEXT,         -- OK / PARTIAL / FAILED
  parse_note   TEXT,
  excluded_by_revenue INTEGER DEFAULT 0
);

-- API 호출량 카운터
CREATE TABLE api_usage (date TEXT PRIMARY KEY, call_count INTEGER);

-- 회사×회계연도 재무 이력 (STEP 7, 2026-07-15 추가 — §4-6 참고)
-- results의 _cur/_prv("가장 최근 감사보고서의 당기·전기")는 그대로 두고,
-- 이 테이블이 "연도별 이력"을 별도로 보관한다.
CREATE TABLE financial_snapshots (
  id                INTEGER PRIMARY KEY AUTOINCREMENT,
  result_id         INTEGER REFERENCES results(id),
  rcept_no          TEXT,   -- 이 연도 수치의 출처 공시
  fiscal_year       TEXT,   -- 회계연도, 예: "2023"
  -- 표준 재무 13항목 (results의 _cur/_prv와 동일 필드셋, 접미어 없음)
  current_assets    INTEGER, noncurrent_assets INTEGER, total_assets INTEGER,
  current_liab      INTEGER, noncurrent_liab   INTEGER, total_liab   INTEGER,
  total_equity      INTEGER,
  revenue           INTEGER, cogs               INTEGER, gross_margin REAL,
  sga               INTEGER, operating_income   INTEGER, net_income   INTEGER,
  parse_status      TEXT,   -- OK / PARTIAL / FAILED
  parse_note        TEXT,
  UNIQUE(result_id, fiscal_year)
);
```

## 6. 백엔드 API 설계
