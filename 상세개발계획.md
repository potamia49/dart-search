# 상세개발계획: 범용 DART 정보 검색기 (웹앱)

> 기준 문서: [PRD.md](PRD.md) — Phase 1 (데이터 수집기) 범위를 **웹앱**으로 구현한다.
> 작성일: 2026-07-14

---

## 1. 개요 및 목표

- 지역 / 매출액 범위 / 업종 조건을 **웹 화면에서 입력**하면, OpenDART API 기반으로
  외부감사대상 법인의 기본정보 + 요약 재무정보(당기·전기)를 수집하여
  **화면 테이블로 조회 + CSV/Excel 다운로드**까지 제공하는 로컬 웹앱.
- 수집 작업은 수 분~수 시간이 걸릴 수 있으므로 **백그라운드 작업(Job) 구조**로 설계하고,
  진행률을 화면에서 실시간 확인할 수 있게 한다.
- 단일 사용자 로컬 실행(`localhost`)을 기본 전제로 하되, 추후 사내 서버 배포가 가능한 구조로 만든다.

### Phase 1 완료 기준 (PRD 8장 → 웹앱 기준으로 재정의)

| # | 완료 기준 |
|---|---|
| 1 | 웹 화면에서 지역/매출액/업종 조건 입력 및 검색 실행 가능 |
| 2 | 조건에 맞는 회사 목록 자동 추출 (백그라운드 Job) |
| 3 | 회사별 기본정보 7항목 수집 (업체명/주소/전화번호/대표자/업종명·코드/결산기준일/감사의견) |
| 4 | 회사별 당기·전기 요약 재무정보 13항목 파싱 (PRD 3-2절 전 항목) |
| 5 | 결과 화면 조회 + CSV/Excel 다운로드 |
| 6 | 진행률/오류 현황 표시 및 실패 건 재시도 기능 |
| 7 | 샘플 10개사 수동 검수로 파싱 정확도 확인 |

| 보조 데이터원 | **공공데이터포털 금융위원회_기업기본정보 API** | 지역 사전 필터용 주소 DB 일괄 구축 (4-1절) — DART의 지역 검색 부재를 보완 |

## 4. 데이터 수집 파이프라인 상세

PRD 5장의 8단계를 Job 실행 단계(STEP)로 매핑한다. 각 STEP 완료 시 DB에 체크포인트를
저장하여 **중단 후 이어하기(resume)** 가 가능해야 한다 (일일 호출한도 20,000건 대응).

> **⚠ 재설계 완료 (설계 2026-07-15, 구현 2026-07-15, A1 전수 크롤 완주
> 2026-07-17) — 아래 STEP 테이블은 M2~M3 시점의 단일 순차 파이프라인
> 설계이며, 더 이상 Job 생성 시 기본 경로가 아니다.** M5에서 STEP 3 처리
> 속도(초당 0.22건, 넓은 기간 조건 시 며칠 소요) 병목이 실측되어,
> **§4-7 "아키텍처 재설계: 2단계 파이프라인 분리"**에서 DART 의존도를
> 낮춘 새 설계(Phase 1 A1~A4 / Phase 2 B1~B5)로 교체했다 — §8 M6 참고.
> 아래 STEP 1~3(corp_cache 갱신·list.json 수집·company.json 지역필터)은
> Phase 1(A1~A4)로 대체됐고, STEP 4~7(원문 다운로드·파싱·매출필터·이력수집)은
> 함수 재작성 없이 Phase 2(B1~B5)가 그대로 재사용한다. 실전 조건(경남
> 김해시, 매출 60~150억) Job #14가 새 파이프라인으로 2026-07-17 완주했다.

| STEP | 내용 | 사용 API | 산출 |
|---|---|---|---|
| 0 | 조건 입력 검증, Job 생성 | - | `jobs` 레코드 |
| 1 | corpCode 캐시 확인/갱신 (7일 경과 시 재다운로드) | corpCode.xml | `corp_cache` 테이블 |
| 2 | 대상 기간 내 "외부감사관련(pblntf_ty=F)" 공시 목록 수집 (페이징 순회) | list.json | 후보 회사 목록 + 접수번호(rcept_no) |
| 3 | **지역 사전 추림**(금융위 주소 DB 매칭, 4-1절 대응 1) → 추려진 후보만 기업개황 조회로 확정 + `corp_profiles` 캐시 적재 | 금융위 기업기본정보, company.json | 필터 통과 회사 목록 |
| 4 | 감사보고서 원본 다운로드 (zip 해제 → XML/PDF 판별) | document.xml | 원문 파일 (로컬 캐시 디렉터리) |
| 5 | 재무제표 파싱 (당기·전기 13항목) + 감사의견 추출 | - | `results` 레코드 (파싱 상태 포함) |
| 6 | **매출액 범위 필터** 적용, 결과 확정 | - | 최종 결과셋 |
| 7 | **최근 N년 재무이력 수집**(`excluded_by_revenue=0`인 최종 결과만 대상, 2026-07-15 추가) | list.json(corp_code 지정) + document.xml | `financial_snapshots` 레코드 (회사×회계연도) |

> **STEP 7 구현 메모 (2026-07-15 추가 — §4-6 참고)**: 사용자가 "최근 N년치
> 재무 이력"을 요청해 STEP 6 뒤에 추가했다. STEP 3(FSC 사전 추림)와 같은
> "값비싼 호출은 후보를 최대한 추린 뒤에" 원칙을 재사용해, STEP 1~6을 다
> 통과한 최종 결과만 대상으로 한다. Job의 DONE 판정 시점이 STEP 6에서 STEP 7
> 완료 시점으로 다시 이동했다.

> **M2 구현 메모 (resume의 STEP별 의미 차이)**: §5 스키마에는 STEP 2(공시 목록) 조회
> 결과를 위한 전용 테이블이 없다. 그래서 STEP 1(corp_cache TTL 체크)과 STEP 2
> (list.json 페이징)는 resume 시 **항상 처음부터 다시 실행**한다 — 둘 다 멱등이고
> 호출 비용이 낮기 때문(STEP1은 TTL 이내면 즉시 skip, STEP2는 페이지 수만큼만 호출).
> 실제로 호출 비용이 큰 STEP 3(company.json, 후보 수만큼)과 STEP 4(document.xml,
> 필터 통과 건수만큼)만 각각 `corp_profiles` TTL 캐시와 `DOCUMENT_CACHE_DIR` 로컬
> 파일 캐시로 "이미 처리된 건은 재호출하지 않는" 방식의 진짜 resume을 구현한다.
> `results` 테이블도 이미 삽입된 corp_code는 재삽입하지 않는다
> (`app/core/pipeline.py`).

### 4-1. 지역 필터 — 구조적 제약과 전역 캐시 전략 ★

> **M2 구현 현황(2026-07-15 갱신)**: 대응 1(금융위 API)의 커버리지 스파이크가
> 완료되어(경남 표본 19/19 매칭, 100%) **대응 1을 채택·구현했다.**
> `app/core/pipeline.py`의 `_resolve_candidate_profile()`이 캐시 미스 시
> `FscCorpInfoClient.get_corp_basic_info()`로 회사명을 먼저 조회해 지역이
> 명백히 다르면 DART `company.json` 호출을 생략하도록 바뀌었다(아래 "결합
> 흐름(실제 구현)" 참고). STEP 3 호출부(`_run_region_industry_filter`)의
> 나머지 로직(지역/업종 매칭, results 선삽입)은 그대로 유지했다. 별도의
> 추상클래스나 플러그인 시스템은 과설계로 판단해 도입하지 않았다.
>
> **원래 계획(아래 "결합 흐름" 1~3단계, §8 M2 체크리스트)과의 차이**: 당초에는
> 금융위 API로 **전체 주소 DB를 사전 일괄 구축**(4만 개사 ≈ 400회 호출)한 뒤
> 회사명으로 매칭하는 방식을 상정했다. 실제로는 이를 별도 배치 스크립트로
> 만들지 않고, **Job 실행 중 STEP 3에서 후보 1건마다 캐시 미스 시점에
> 그때그때 FSC 회사명 조회 1건을 먼저 호출**하는 방식으로 단순화했다 — 결과는
> 동일(지역이 명백히 다른 후보의 DART 호출 생략)하지만 사전 일괄 적재
> 스크립트/전용 테이블이 없어도 되므로 구현이 더 단순하고, `corp_profiles`가
> Job을 거치며 점진적으로 채워지는 기존 대응 2의 방식과도 일관된다(과설계
> 방지). 새 DB 테이블은 추가하지 않았다 — `corp_profiles`에 FSC로만 확인된
> 부분 데이터(`sido`/`sigungu`만 채우고 `phone`/`ceo_name`/`induty_code`는
> `NULL`)를 upsert하는 것으로 충분했다.

**제약**: OpenDART API에는 지역 검색 파라미터가 없다. 회사의 주소는 기업개황
(company.json)을 **회사별 1건씩 조회**해야만 알 수 있다. 따라서 "김해만 검색"하더라도
어느 회사가 김해인지 판별하기 위해 후보 전체(대상 기간 내 감사보고서 제출 회사,
연간 약 3~4만 개사)의 기업개황이 필요하며, **최초 실행 비용은 김해 단독이든 경남
전체든 동일**하다 (일일 한도 20,000건 → 최초 1회 약 2일 소요).

**대응 1 (채택, 2026-07-15) — 금융위원회_기업기본정보 API로 지역 사전 추림**:
- 공공데이터포털(data.go.kr)의 **금융위원회_기업기본정보조회 API**(`getCorpOutline_V2`,
  무료, 별도 키 발급)는 DART 공시법인의 기본정보(본점 주소 `enpBsadr` 포함)를
  회사명(`corpNm`) 조회로 확인 가능하고, OpenDART와 **별도 쿼터**를 사용한다
  (`app/core/dart_client.py`의 `FscCorpInfoClient`, `DartClient`의 `api_usage`
  카운터와 무관).
- 원래 계획한 흐름(참고용, 실제 구현과는 아래 "실제 구현" 문단 참고):
  1. STEP 2 (DART list.json)로 후보 회사 목록 확보 — 기존과 동일
  2. 금융위 주소 DB와 **회사명 매칭** → 지역 후보만 추림 (3~4만 → 수백 개사)
     - 매칭 전 이름 정규화: "(주)/주식회사" 제거, 공백/괄호 통일 (`filters.normalize_corp_name`)
  3. 추려진 후보만 DART company.json 호출 → 주소 최종 확정(오탐 제거) +
     전화번호·대표자 수집 (Phase 1 필수 수집 항목이므로 낭비 호출 아님)
- **✅ 스파이크 실행 완료 (2026-07-15)**: 경상남도 소재 비상장 외감법인 표본 19건
  (목표 20건) 전수 매칭, **커버리지율 100%** → **대응 1 채택 확정**. 실행 스크립트는
  `backend/app/core/spike_financial_committee_coverage.py`. 응답 스키마(`response.body.
  items.item`이 리스트, 결과 없음일 때도 빈 리스트를 감싼 dict로 와서 truthy 판정만으로는

> **M3 구현 현황(2026-07-15) — 실측 메모**: 실제 DART API로 원문 30건을
> 확보해(2026년 4~6월 접수분 25건 + 2012년 초 접수분 5건,

> 마주치지 못했을 뿐 존재 자체가 없다는 뜻은 아니므로, `pdf_parser.py`는
> best-effort로 구현해 두었고 M5에서 실물이 나오면 검증한다). 실측으로 확인된
> 원문 구조와 서식 편차(회사마다 로마숫자 표기가 유니코드/아스키로 갈리거나,
> 각주 참조가 "(주석13)"/"(주6)"로 축약 방식이 다르거나, 매출원가/판관비를

1. **1순위 — XML 파싱**: document.xml이 반환하는 원본 zip에는 DART 접수 XML이
   포함되는 경우가 대부분. `<TABLE>` 구조에서 재무상태표/손익계산서 표를 탐색.
   실제 구현(`app/parsers/xml_parser.py`)은 `<TITLE>` 텍스트로 재무상태표/
   손익계산서 구간을 판별한 뒤 그 구간의 첫 `ACLASS="FINANCE"` 테이블만
   사용한다(자본변동표/현금흐름표도 같은 ACLASS를 쓰므로 혼입 방지). 원문
   본문에 이스케이프 안 된 특수문자가 섞여 엄격 XML 파싱이 실패하는 경우가
   실측 30건 중 다수 있어 `lxml.etree.XMLParser(recover=True)`로 손상된
   부분만 건너뛰고 나머지 구조를 살린다.
2. **2순위 — PDF 파싱**: 원본이 PDF인 경우 `pdfplumber`로 표 추출
   (`app/parsers/pdf_parser.py` — 위 실측 메모대로 실제 PDF 표본으로
   검증되지 않은 best-effort 구현).
3. **HWP**: Phase 1에서는 파싱 시도하되 실패 시 `PARSE_FAILED(HWP)` 상태로 기록만
   하고 넘어간다 (수동 검수 대상 목록에 포함). 미구현 상태 유지.
4. **계정과목 정규화 사전**: "매출액/영업수익/수익(매출액)", "당기순이익/당기순손실"
   등 표기 변형 → 표준 13항목으로 매핑하는 사전을 `parsers/base.py`에 정의하고,
   검수 과정에서 지속 보강한다. "영업손실"/"당기순손실"처럼 손실이 명시된
   행은 원문 숫자가 부호 없이 양수로 찍혀 있어 저장 시 부호를 뒤집는다.
   "매출총이익율(%)"은 원문의 "매출총이익"(금액) 행을 그대로 쓰지 않고
   매출액/매출원가로부터 계산한다(PRD 3-2절 정의, `compute_gross_margin()`).
5. 당기/전기 컬럼 판별: 헤더의 "제 N 기" 숫자 비교 또는 좌→우 순서 규칙 적용.
   실제 구현은 THEAD의 TH `COLSPAN`으로 당기/전기 열 개수를 판별하고, 각
   TR의 값 셀을 그 열 개수만큼 앞/뒤로 나눠 그룹 내 첫 번째 비어있지 않은
   셀을 채택한다(같은 그룹 안에 "상세/합계" 두 하위 열이 있는 서식이 실측
   대부분이었다).
6. 모든 파싱 결과에 `parse_status`(OK / PARTIAL / FAILED)와 `parse_note`를 남겨
   화면에서 수동 검수 대상을 바로 필터링할 수 있게 한다. 실측 기준: 재무제표
   자체가 첨부되지 않은 경우(의견거절 등, ACLASS="FINANCE" 테이블 없음)와
   서비스업 서식이라 매출원가/판관비 항목이 구조적으로 없는 경우 모두
   PARTIAL로 판정하고(파싱 실패가 아니라 원문에 없는 것), XML 구문이 복구
   불가할 정도로 손상된 경우만 FAILED로 판정한다.

   일치**(FSC 값이 "천원" 단위로 저장돼 마지막 3자리만 절삭되는 차이만
   있었다).
2. **다년치 이력 불가**: 같은 회사로 전기(`bizYear` 1년 전)를 조회하면
   `totalCount=0` — **이 API는 최신 연도 1개년만 보유**한다. STEP 7
   (다년치 이력, §4-6)을 대체할 수 없고, "당기 스크리닝" 용도로만 쓸 수
   있다.
3. **`getBs_V2`/`getIncoStat_V2`(재무상태표/손익계산서 오퍼레이션) 미확인**:
   호출은 `resultCode=00`(정상)으로 응답했지만 `totalCount=0`이었다 —
   오퍼레이션명 자체가 틀렸을 가능성이 있어 이번 세션에서는 결론 내지
   못했다(cogs/sga 사전 확보 가능성은 미확정 TODO, 아래 "열린 질문 3" 참고).
4. **`fssCorpUnqNo` = DART `corp_code` 100% 일치 (5/5 검증)**: 금융위원회_
   기업기본정보(`getCorpOutline_V2`) 응답의 `fssCorpUnqNo` 필드가 DART의
   8자리 `corp_code`와 정확히 같은 값이었다 — 기존 §4-1 "대응 1"이 쓰는
   회사명 정규화 매칭(`normalize_corp_name`)보다 훨씬 신뢰도 높은 직접
   조인 키다. 단, 같은 회사라도 소스기관(금감원/산업은행/신용보증기금/
   기업은행 — 아래 6번 FAQ 근거)별 중복 레코드가 여러 건 응답에 섞여 오고,
   그중 일부 레코드에만 `fssCorpUnqNo`가 채워져 있었다(실측 예시: `유진
   금속공업(주)` 3건 중 2건은 `fssCorpUnqNo="00567444"`, 1건은 빈 문자열이지만
   대신 `sicNm="알루미늄주물 주조업"`을 갖고 있어 서로 다른 레코드가 서로
   다른 필드를 보완적으로 채우는 구조) — 이 4가지 raw 응답 필드
   목록(`crno`/`corpNm`/`corpEnsnNm`/`enpRprFnm`(대표자명)/`bzno`(사업자등록번호)/
   `enpBsadr`/`enpDtadr`(주소)/`enpHmpgUrl`/`enpTlno`/`enpFxno`(전화/팩스)/
   `sicNm`(업종명, 코드 없음)/`enpEstbDt`(설립일)/`enpStacMm`(결산월)/
   `enpEmpeCnt`(종업원수)/`actnAudpnNm`(감사인명)/`audtRptOpnnCtt`(감사의견
   내용, 이번 표본에서는 비어 있었음)/`fssCorpUnqNo`)도 이번 스파이크로
   실측 확인했다.
5. **전체 모수 확인**: `corpNm` 없이 `getCorpOutline_V2`를 호출하면(=이름
   필터 없이 전체 페이징) **`totalCount=1,282,065`** — 원래 §4-1이 가정했던
   "4만 개사·약 400회 호출" 추정보다 훨씬 크다. `numOfRows=100` 기준 전체
   순회에 약 12,800회 호출이 필요하다(실제 소요 시간은 미검증, 아래 "열린
   질문 1" 참고).
6. **외부감사여부 필드 없음 (공식 FAQ 확인, `fsc.go.kr/in060501` Q11)**:
   "비외감법인 데이터만을 따로 구분하여 조회할 수 없으며... 외감법인과
   비외감법인을 모두 포함한 데이터를 조회할 수 있습니다"라고 명시돼 있다
   — 다만 이 프로젝트는 STEP 2(`pblntf_ty=F`)로 이미 외감법인만 다루므로
   무관하다. 같은 FAQ의 Q8은 데이터 원천이 "금융표준종합정보시스템"이
   금융감독원·산업은행·신용보증기금·기업은행 등 9개 산하기관에서 수집한다고
   설명한다(외감대상은 금감원 데이터 우선, 비외감대상은 중소기업은행 데이터
   우선 — 위 4번의 중복 레코드 현상과 일치).

**STEP 상세**:

- **A1 (FSC 전역 인덱스 구축)**: `FscCorpInfoClient.get_corp_basic_info()`를
  `corp_nm` 없이 페이징 순회(전체 1,282,065건, `numOfRows=100` 기준 약

  레코드는 **`crno` 기준으로 그룹핑한 뒤 필드별로 존재하는 값을 채택하는

  레코드에만, `sicNm`은 다른 레코드에만 채워지는 식. 실측 예시: 유진금속공업
  중복 3건 모두 `crno="1846110041115"` 동일, `fssCorpUnqNo`는 그중 2건만
  채워짐 — 병합 기준 키를 `fssCorpUnqNo`가 아니라 `crno`로 잡아야 하는 이유,
  아래 "PK 재검증" 참고). 갱신 주기는 `corp_profiles`(대응 2)와 같은 철학으로
  TTL 180일 — 매 Job마다 다시 돌지 않는다.
  **PK 설계 재검증 및 A1 파일럿 실측 (2026-07-15, 실제 API 호출로 확인)**:
  - `fss_corp_unq_no`는 PK로 쓸 수 없다 — 무작위 표본(이름 필터 없이 1페이지,
    n=100)에서 24%가 빈 문자열이었다. `crno`도 100% 신뢰할 수는 없다 —
    해외 레코드(예: `LEHMAN BROTHERS INC`, `ARMOR QUALIFIED, LP` 등
    `corpRegMrktDcd="E"`(기타/외국계) 레코드)는 `crno="0000000000000"`
    더미값이 채워진다. 다만 이런 해외 레코드는 국내 주소 형식이 아니므로
    A2(주소 기반 지역 필터)에서 자연히 탈락하니 기능상 문제는 없다 —
    다만 저장 효율을 위해 `corpRegMrktDcd='E'`인 레코드는 A1 적재 시점에
    걸러도 된다(선택적 최적화). §5의 `fsc_corp_index` 스키마를 실제 PK
    `id`(AUTOINCREMENT) + `crno`/`fss_corp_unq_no` 부분 UNIQUE 인덱스(더미/
    빈값 제외)로 정정했다.
  - **50페이지 파일럿 실측**: 순차 호출 평균 2.875초/페이지(네트워크 포함,
    동시성 없음). `totalCount=1,282,065`(재검증으로 재확인) 기준 전체

    해소). data.go.kr 자체 rate limit에 걸리는 현상은 50페이지 동안
    없었다. 동시성을 도입하면(기존 STEP3의 동시 5건 제한 패턴 재사용 가능)
    단축 가능하나 data.go.kr의 동시 접속 정책은 별도 검증 필요 — 처음
    구현은 순차로 하고, 실제 운영 중 필요하면 동시성을 추가하는 방향을
    권장(과설계 방지).
  - **총자산/총부채/총자본 필드 확인**: 별도 오퍼레이션(`getBs_V2`) 없이

    (총자산)/`enpTdbtAmt`(총부채)/`enpTcptAmt`(총자본)/`enpCptlAmt`(자본금)/

    §4-7-2의 "총자산 사전 스크리닝(A3)"가 `getBs_V2` 확인 없이도 바로

   직접법 "영업활동현금흐름" 계열)은 실측 표본엔 없었으나 흔한 표기라
   `CF_ACCOUNT_NAME_ALIASES`에 방어적으로 함께 등록했다. 별도로 Job 문서
   캐시(945건+)까지 확장 실측하지는 않았다 — fixtures 19건이 이미 100% 일관
   되고 산식 접미어 케이스까지 커버했기 때문(추가 실측은 과투자로 판단).
2. **기존 완료 Job 소급 재파싱은 하지 않는다(확정)** — 신규 Job 실행분부터만
   CF를 채운다. 관리자용 "전체 재파싱" 트리거는 만들지 않았다(별도 도구를
   추가할 만큼 급하지 않고, CF는 best-effort 항목이라 기존 Job의 CF 공란이
   기능 결함이 아니다). 실측 확인: Job #18(기존 DONE)의 result는 `cf_*_cur/prv`가
   전부 NULL로 남아 있고(재파싱 안 함), **원문 섹션 열람(③④)은 이와 무관하게
   기존 Job에서도 즉시 동작한다** — 로컬 문서 캐시에서 on-demand로 잘라내므로
   DB에 CF 수치가 없어도 원문 현금흐름표를 그대로 볼 수 있다(실측 확인).
3. **원문 손상 문서 표시** — 섹션 추출은 "섹션 마크에 해당하는 첫 `<TITLE>`의
   부모 컨테이너(`TABLE-GROUP`/`SECTION-*`)를 통째로 렌더링"하는 방식으로
   구현했다(실측 구조: BS/IS/CF는 `TABLE-GROUP`, 주석은 `SECTION`에 담김).
   `recover=True`로 잘린 문서는 잘린 그대로 렌더링되며(에러 아님), 섹션 자체를
   못 찾으면 `available=false`+안내 문구를 반환한다. 별도 "원문 일부 손상"
   배지는 추가하지 않았다(잘린 경우와 정상 경우를 원문 파싱만으로 구분하기
   어렵고, 사용자에겐 DART 전체 보고서 링크가 항상 병행 제공됨).

**구현 메모(2026-07-19)**:
- §4-8 배경이 CF 예시로 든 `20260630000651`은 실제로는 재무상태표/손익계산서

  못 잡는다 — 예시가 부정확했다. CF 정상 추출은 `20260630000641` 등 19건에서

### 4-9. 비UTF-8(EUC-KR/CP949) 원문 인코딩 폴백 (2026-07-19, 구현 완료)

M5 실전 실행 중 발견한 EUC-KR 원문 1건(Job #14를 통째로 죽였던 건, §4-8 이전
기록 참고)이 얼마나 흔한지 재조사해 근본 복구를 넣었다. 수동 관찰(`parse_note`)
전략은 데이터가 안 쌓여(현재 DB에 실제 재무 파싱 Job이 #18 5건뿐, Job #14는
리셋됨) 결론이 안 나던 상태였으므로, **이미 받아둔 로컬 문서 캐시
(`data/documents`, 1,453건)를 직접 스캔**해 실측했다(쿼터 0):
- **1,453건 전부 XML 선언부는 `encoding="utf-8"`인데, 실제 바이트가 EUC-KR/CP949인
  원문이 64건(약 4.4%)** — 체계적 패턴이며 선언부가 거짓이라 선언부로는 못 잡는다.
  64건 전부 CP949 디코딩 시 복구(하드 실패 0), 그 중 51건은 재무 테이블까지 복구.
  기존 코드는 이 51건을 전부 FAILED로 버리고 있었다.
- **수정**: `xml_parser.py::_decode_raw_xml()` — 파싱 진입 전 인코딩을 UTF-8로
  정규화. UTF-8 성공 시 바이트 그대로 통과(정상 원문 95.6% 무변경), `UnicodeDecodeError`일
  때만 CP949 폴백 → 거짓 인코딩 선언부 제거 → UTF-8 재인코딩(둘 다 실패 시
  `errors="replace"`). `recover=True`가 인코딩 오류를 복구 못 하는 이유: 인코딩
  오류는 XML 구조 오류가 아니라 파싱 진입 단계의 fatal error라서.
  `determine_parse_status()`·계정과목 사전은 무변경.
- **소급 재파싱은 안 함**(§4-8 CF 소급 미시행과 동일 원칙) — 신규 Job부터 자동 복구.
- 회귀 테스트: 실측 EUC-KR 원문(rcept_no=20220127000408, 남경산업)을 fixtures에
  추가, `_decode_raw_xml` 유닛 2종 + `parse_status=OK` 복구 검증 1종. `pytest -q`
  **183 passed**(180 + 3).
