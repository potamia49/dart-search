# 상세개발계획: 범용 DART 정보 검색기 (웹앱)

> 기준 문서: [PRD.md](PRD.md) — Phase 1 (데이터 수집기) 범위를 **웹앱**으로 구현한다.
> 작성일: 2026-07-14

---

## 1. 개요 및 목표

- 지역 / 매출액 범위 / 업종 조건을 **웹 화면에서 입력**하면, OpenDART API 기반으로
  외부감사대상 법인의 기본정보 + 요약 재무정보(당기·전기)를 수집하여
  **화면 테이블로 조회 + CSV/Excel 다운로드**까지 제공하는 로컬 웹앱.
- 수집 작업은 수 분~수 시간이 걸릴 수 있으므로 **백그라운드 작업(Job) 구조**로 설계하고,
  진행률을 화면에서 실시간 확인할 수 있게 한다.
- 단일 사용자 로컬 실행(`localhost`)을 기본 전제로 하되, 추후 사내 서버 배포가 가능한 구조로 만든다.

### Phase 1 완료 기준 (PRD 8장 → 웹앱 기준으로 재정의)

| # | 완료 기준 |
|---|---|
| 1 | 웹 화면에서 지역/매출액/업종 조건 입력 및 검색 실행 가능 |
| 2 | 조건에 맞는 회사 목록 자동 추출 (백그라운드 Job) |
| 3 | 회사별 기본정보 7항목 수집 (업체명/주소/전화번호/대표자/업종명·코드/결산기준일/감사의견) |
| 4 | 회사별 당기·전기 요약 재무정보 13항목 파싱 (PRD 3-2절 전 항목) |
| 5 | 결과 화면 조회 + CSV/Excel 다운로드 |
| 6 | 진행률/오류 현황 표시 및 실패 건 재시도 기능 |
| 7 | 샘플 10개사 수동 검수로 파싱 정확도 확인 |

## 2. 기술 스택

| 구분 | 선택 | 사유 |
|---|---|---|
| 백엔드 | **Python 3.12 + FastAPI** | PDF/HWP 파싱 생태계가 Python에 집중(pdfplumber 등). API 서버와 수집 로직을 한 언어로 통일 |
| 백그라운드 작업 | FastAPI `BackgroundTasks` → 규모 커지면 자체 워커 스레드 | Phase 1은 단일 사용자라 Celery/Redis 등 외부 인프라 불필요 |
| DB | **SQLite** (`sqlite3` + SQLAlchemy) | 로컬 단일 파일, corpCode 캐시·수집결과·Job 상태 저장. 배포 시 PostgreSQL 전환 가능 구조 |
| 프론트엔드 | **React 18 + Vite + TypeScript** | 조건 입력 폼 + 진행률 + 결과 테이블 SPA. UI 라이브러리는 Mantine(또는 shadcn/ui) |
| HTTP 클라이언트 | `httpx` (비동기) | OpenDART API 병렬 호출 + 타임아웃/재시도 제어 |
| 파싱 | XML 파서(표준 `lxml`) 우선, PDF는 `pdfplumber`, HWP는 `olefile`+`hwp5` 계열 | 감사보고서 원본은 XML이 다수 → 4-4절 참고 |
| Excel 출력 | `openpyxl` + `pandas` | |
| 환경변수 | `.env` + `pydantic-settings` | OpenDART·공공데이터포털 API 키 관리, 코드 하드코딩 금지 |
| 보조 데이터원 | **공공데이터포털 금융위원회_기업기본정보 API** | 지역 사전 필터용 주소 DB 일괄 구축 (4-1절) — DART의 지역 검색 부재를 보완 |

## 4. 데이터 수집 파이프라인 상세

PRD 5장의 8단계를 Job 실행 단계(STEP)로 매핑한다. 각 STEP 완료 시 DB에 체크포인트를
저장하여 **중단 후 이어하기(resume)** 가 가능해야 한다 (일일 호출한도 20,000건 대응).

| STEP | 내용 | 사용 API | 산출 |
|---|---|---|---|
| 0 | 조건 입력 검증, Job 생성 | - | `jobs` 레코드 |
| 1 | corpCode 캐시 확인/갱신 (7일 경과 시 재다운로드) | corpCode.xml | `corp_cache` 테이블 |
| 2 | 대상 기간 내 "외부감사관련(pblntf_ty=F)" 공시 목록 수집 (페이징 순회) | list.json | 후보 회사 목록 + 접수번호(rcept_no) |
| 3 | **지역 사전 추림**(금융위 주소 DB 매칭, 4-1절 대응 1) → 추려진 후보만 기업개황 조회로 확정 + `corp_profiles` 캐시 적재 | 금융위 기업기본정보, company.json | 필터 통과 회사 목록 |
| 4 | 감사보고서 원본 다운로드 (zip 해제 → XML/PDF 판별) | document.xml | 원문 파일 (로컬 캐시 디렉터리) |
| 5 | 재무제표 파싱 (당기·전기 13항목) + 감사의견 추출 | - | `results` 레코드 (파싱 상태 포함) |
| 6 | **매출액 범위 필터** 적용, 결과 확정 | - | 최종 결과셋 |

### 4-1. 지역 필터 — 구조적 제약과 전역 캐시 전략 ★

**제약**: OpenDART API에는 지역 검색 파라미터가 없다. 회사의 주소는 기업개황
(company.json)을 **회사별 1건씩 조회**해야만 알 수 있다. 따라서 "김해만 검색"하더라도
어느 회사가 김해인지 판별하기 위해 후보 전체(대상 기간 내 감사보고서 제출 회사,
연간 약 3~4만 개사)의 기업개황이 필요하며, **최초 실행 비용은 김해 단독이든 경남
전체든 동일**하다 (일일 한도 20,000건 → 최초 1회 약 2일 소요).

**대응 1 (우선 검증) — 금융위원회_기업기본정보 API로 주소 DB 일괄 구축**:
- 공공데이터포털(data.go.kr)의 **금융위원회_기업기본정보조회 API**(무료, 별도 키 발급)는
  DART 공시법인의 기본정보(본점 주소, 법인등록번호 포함)를 **페이지네이션 일괄 수집**
  가능 (1회 최대 100건 → 4만 개사 ≈ 400회 호출, 수 시간 내 완료).
- 결합 흐름:
  1. STEP 2 (DART list.json)로 후보 회사 목록 확보 — 기존과 동일
  2. 금융위 주소 DB와 **회사명 매칭** → 지역 후보만 추림 (3~4만 → 수백 개사)
     - 매칭 전 이름 정규화: "(주)/주식회사" 제거, 공백/괄호 통일
     - 법인등록번호 앞 4자리(관할 등기소 코드)로 주소 결측 건 보조 판별
  3. 추려진 후보만 DART company.json 호출 → 주소 최종 확정(오탐 제거) +
     전화번호·대표자 수집 (Phase 1 필수 수집 항목이므로 낭비 호출 아님)
- **검증 필요(스파이크, M1에 반나절 배정)**: 소형 외감법인(감사보고서만 제출)까지
  커버하는지 실측. 커버리지 미달 시 아래 대응 2로 폴백.
- 검토 후 제외한 대안: DART 홈페이지 지역검색 크롤링(약관 위반, PRD 제외),
  중소기업현황정보시스템 sminfo(지역 검색 가능하나 자동화용 API 미제공),
  크레탑/KISLINE 등 상용 DB(확실하나 유료 — 필요 시 Phase 2에서 재검토).

**대응 2 (폴백) — `corp_profiles` 전역 캐시**:
- 기업개황 조회 결과(주소/시도/시군구/업종코드/대표자/전화)를 Job과 무관한
  **전역 마스터 테이블**에 영구 저장. 한 번 조회한 회사는 재조회하지 않는다.
- 이후 검색(김해 → 양산 → 창원 등 조건 변경 재검색)은 캐시만으로 즉시 지역 필터링
  → STEP 3의 API 호출이 신규 회사분만으로 급감.
- 주소·업종은 변동이 드물므로 갱신 주기는 길게(기본 180일, 설정 가능).
- 캐시 적재 시 주소에서 시도/시군구를 파싱해 별도 컬럼으로 저장 → 필터는 컬럼
  매칭으로 수행 (매번 문자열 검색하지 않음).
- "예상 규모 미리보기"에서 **캐시 보유율**(예: 후보 32,000개사 중 캐시 28,000건,
  신규 호출 필요 4,000건)을 함께 표시해 사용자가 소요 시간을 예측할 수 있게 한다.

**필터 로직**:
- 프론트에서 **광역시도 선택 → 시/군/구 다중 선택** 2단계 UI 제공.
- 주소 표기 편차 대응: "경상남도/경남" 등 약칭 매핑 테이블을 `filters.py`에 상수로 정의.

### 4-2. 업종 필터

- 기업개황 `induty_code`(표준산업분류) prefix 매칭.
- 프론트에서 대분류 → 중분류 트리 선택 UI (복수 선택). 분류코드 마스터는
  통계청 KSIC 10차 목록을 JSON으로 백엔드에 내장(`meta.py`에서 서빙).

### 4-3. 매출액 필터

- **사전 필터 불가**: 비상장 외감법인은 재무제표 API가 지원되지 않아(PRD 4장),
  원문을 파싱하기 전에는 매출액을 알 수 없다. 따라서 매출액은 구조적으로
  **마지막 단계의 사후 필터**일 수밖에 없다. 대신 지역·업종 필터(STEP 3)를 먼저
  통과시켜 다운로드·파싱 대상을 최소화한다.
- 파싱된 당기 매출액 기준 `min ≤ 매출액 ≤ max` (억원 단위 입력 → 원 단위 환산).
- 사후 필터이므로 필터 탈락 건도 `results`에 남기되 `excluded_by_revenue` 플래그 처리
  (재검색 시 재파싱 없이 재사용).

### 4-4. 재무제표 파싱 전략 (핵심 리스크 구간)

1. **1순위 — XML 파싱**: document.xml이 반환하는 원본 zip에는 DART 접수 XML이
   포함되는 경우가 대부분. `<TABLE>` 구조에서 재무상태표/손익계산서 표를 탐색.
2. **2순위 — PDF 파싱**: 원본이 PDF인 경우 `pdfplumber`로 표 추출.
3. **HWP**: Phase 1에서는 파싱 시도하되 실패 시 `PARSE_FAILED(HWP)` 상태로 기록만
   하고 넘어간다 (수동 검수 대상 목록에 포함).
4. **계정과목 정규화 사전**: "매출액/영업수익/수익(매출액)", "당기순이익/당기순손실"
   등 표기 변형 → 표준 13항목으로 매핑하는 사전을 `parsers/base.py`에 정의하고,
   검수 과정에서 지속 보강한다.
5. 당기/전기 컬럼 판별: 헤더의 "제 N 기" 숫자 비교 또는 좌→우 순서 규칙 적용.
6. 모든 파싱 결과에 `parse_status`(OK / PARTIAL / FAILED)와 `parse_note`를 남겨
   화면에서 수동 검수 대상을 바로 필터링할 수 있게 한다.

### 4-5. API 호출량 관리

- `dart_client.py`에 일일 호출 카운터(DB 저장) 내장. 상한(기본 19,000, 여유분 확보)
  도달 시 Job을 `PAUSED_QUOTA` 상태로 자동 전환하고 다음 날 재개 버튼 제공.
- 요청 간 딜레이(기본 0.1초) 및 429/5xx 지수 백오프 재시도(최대 3회).

## 5. DB 스키마 (SQLite)

```sql
-- corpCode 캐시
CREATE TABLE corp_cache (
  corp_code   TEXT PRIMARY KEY,   -- 8자리 고유번호
  corp_name   TEXT NOT NULL,
  stock_code  TEXT,               -- 상장사만 존재 (비상장 필터에 활용)
  modify_date TEXT
);
CREATE TABLE cache_meta (key TEXT PRIMARY KEY, value TEXT);  -- corp_cache 갱신일 등

-- 기업개황 전역 캐시 (4-1절 — Job과 무관하게 영구 보존, 재검색 시 재사용)
CREATE TABLE corp_profiles (
  corp_code   TEXT PRIMARY KEY,
  corp_name   TEXT, address TEXT,
  sido        TEXT, sigungu TEXT,   -- 주소에서 파싱해 저장 → 지역 필터는 컬럼 매칭
  induty_code TEXT, phone TEXT, ceo_name TEXT,
  fetched_at  TEXT                  -- 갱신 주기 판단용 (기본 180일)
);

-- 검색 Job
CREATE TABLE jobs (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  created_at    TEXT, name TEXT,
  cond_region   TEXT,   -- JSON: {"sido":"경남","sigungu":["김해시","양산시"]}
  cond_revenue  TEXT,   -- JSON: {"min_krw":6000000000,"max_krw":15000000000}
  cond_industry TEXT,   -- JSON: ["C25","C29"]
  cond_period   TEXT,   -- JSON: {"bgn_de":"20250101","end_de":"20251231"}
  status        TEXT,   -- PENDING/RUNNING/PAUSED_QUOTA/DONE/FAILED/CANCELLED
  current_step  INTEGER, progress_done INTEGER, progress_total INTEGER,
  error_msg     TEXT
);

-- 수집 결과 (회사 1건 = 1행)
CREATE TABLE results (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  job_id      INTEGER REFERENCES jobs(id),
  corp_code   TEXT, rcept_no TEXT,
  -- 기본정보 (PRD 3-1)
  corp_name   TEXT, address TEXT, phone TEXT, ceo_name TEXT,
  induty_code TEXT, induty_name TEXT,
  fiscal_date TEXT,          -- 결산기준일
  audit_opinion TEXT,        -- 적정/한정/부적정/의견거절
  -- 요약 재무 (PRD 3-2) : 당기(_cur) / 전기(_prv), 단위: 원
  current_assets_cur      INTEGER, current_assets_prv      INTEGER,
  noncurrent_assets_cur   INTEGER, noncurrent_assets_prv   INTEGER,
  total_assets_cur        INTEGER, total_assets_prv        INTEGER,
  current_liab_cur        INTEGER, current_liab_prv        INTEGER,
  noncurrent_liab_cur     INTEGER, noncurrent_liab_prv     INTEGER,
  total_liab_cur          INTEGER, total_liab_prv          INTEGER,
  total_equity_cur        INTEGER, total_equity_prv        INTEGER,
  revenue_cur             INTEGER, revenue_prv             INTEGER,
  cogs_cur                INTEGER, cogs_prv                INTEGER,
  gross_margin_cur        REAL,    gross_margin_prv        REAL,   -- 매출총이익율(%)
  sga_cur                 INTEGER, sga_prv                 INTEGER,
  operating_income_cur    INTEGER, operating_income_prv    INTEGER,
  net_income_cur          INTEGER, net_income_prv          INTEGER,
  -- 상태
  parse_status TEXT,         -- OK / PARTIAL / FAILED
  parse_note   TEXT,
  excluded_by_revenue INTEGER DEFAULT 0
);

-- API 호출량 카운터
CREATE TABLE api_usage (date TEXT PRIMARY KEY, call_count INTEGER);
```

## 7. 화면 설계 (3개 페이지)

### 7-1. 검색 조건 입력 (SearchPage)
- 지역: 시도 셀렉트 → 시군구 멀티 체크박스

  예상 API 호출 M건" 표시 → 사용자가 실행 여부 판단

### 7-2. 작업 현황 (JobsPage)
- Job 카드 목록: 조건 요약, 상태 배지, STEP 진행바 (예: "STEP 5 파싱 중 — 132/480")
- 오늘 API 호출량/잔여량 상단 표시
- PAUSED_QUOTA 상태 Job에 [이어하기] 버튼

### 7-3. 결과 조회 (ResultPage)
- 결과 테이블: 기본정보 + 주요 재무 컬럼 (컬럼 표시/숨김 토글)

- 행 클릭 → 상세 패널 (당기·전기 전 항목 + DART 원문 링크 `dart.fss.or.kr/dsaf001/main.do?rcpNo=...`)
- [Excel 다운로드] / [CSV 다운로드]

| API 키 노출 | `.env` 관리, 프론트에 키 미노출(모든 DART 호출은 백엔드 경유) |

## 10. Phase 2 대비 설계 포인트

- `results` 테이블에 원재무 13항목을 당기·전기 모두 보존 → 파생지표(영업이익률,
  유동비율, 부채비율, 매출액증가율 등)는 Phase 2에서 계산 가능
- 원문 파일 로컬 캐시 유지 → 추가 항목 필요 시 재다운로드 없이 재파싱 가능
- 업종코드 수집 완료 → 업종별 맞춤 코멘트 생성에 바로 활용
- FastAPI 구조이므로 전단지 생성(HTML→PDF) 엔드포인트를 동일 서버에 추가하는
  형태로 자연스럽게 확장
