# dart-search 개발팀 오케스트레이터

dart-search 저장소에서 코드 작업(백엔드/파서/프론트)을 전문 에이전트에게 위임하고,
구현 성격에 맞는 읽기 전용 리뷰 에이전트로 검증까지 마친 뒤 결과를 종합 보고한다.

**실행 모드: 서브 에이전트 패턴.** 이 팀의 에이전트들은 서로 다른 계층(백엔드/파서/
프론트)을 순차적으로 다루고, 구현 다음에 검증이 뒤따르는 "생성 → 검증" 흐름이지
실시간 토론이 필요한 협업이 아니다 — `Agent` 도구로 직접 호출하고 결과를 이 오케스트레이터가
수집·종합한다. 여러 영역이 동시에 걸린 작업이라도 영역별로 순차 호출한다(예: 파서 로직
변경이 백엔드 파이프라인에 영향을 주면 dart-parser 완료 후 dart-backend를 호출).

## Phase 0: 컨텍스트 확인

작업을 시작하기 전에 CLAUDE.md 최상단 근처의 "## 하네스: dart-search 개발팀" 변경
이력과, 이번 대화에서 이미 어떤 에이전트를 호출했는지 확인한다.
- 이번 대화에서 이미 구현 에이전트를 호출한 결과가 있고 사용자가 "그 부분만 다시",
  "QA 반영해서 고쳐줘" 같은 후속 요청을 하면 → **부분 재위임**: 처음부터 다시 계획하지
  말고 이전 결과를 컨텍스트로 전달하며 필요한 에이전트만 다시 호출한다.
- 새 요청이면 → 아래 Phase 1부터 정상 진행한다.

이 프로젝트는 코드베이스이지 생성형 산출물 파이프라인이 아니므로, 별도 `_workspace/`
중간 파일을 두지 않는다 — git이 이미 변경 이력을 추적하고, "산출물"은 각 에이전트가 직접
수정한 코드 그 자체다.

## Phase 1: 작업 분류

요청 내용을 보고 어떤 전문 에이전트가 필요한지 판단한다. 세부 역할 정의는 각
`.claude/agents/{name}.md`를 반드시 먼저 읽고 그 안의 "작업 시작 전 필수"를 따른다.

| 요청 성격 | 위임 대상 |
|---|---|
| FastAPI/DB 스키마/파이프라인(Job, STEP)/OpenDART·공공데이터포털 연동 | `dart-backend` |
| 감사보고서 원문(XML/PDF) 파싱, 계정과목 추출, parse_status 판정 | `dart-parser` |
| React/Vite/Mantine 화면, 폴링 UX, 결과 테이블/Drawer | `dart-frontend` |
| 여러 영역에 걸침 (예: 새 재무 항목 추가 = 파서+백엔드+프론트) | 파서 → 백엔드 → 프론트 순으로 순차 위임 (하위 계층부터 — 프론트가 참조할 데이터 계약이 먼저 확정돼야 한다) |
| 코드 수정 없이 QA/디자인 리뷰만 요청 | Phase 2를 건너뛰고 바로 Phase 3 |

CLAUDE.md는 계속 갱신되는 "현재 진실"이다 — 각 에이전트가 참고할 상세개발계획.md와
실제 최신 상태가 어긋나면 CLAUDE.md 최신 기록이 항상 우선한다는 점을 위임 시 함께
전달한다.

## Phase 2: 구현 위임

```
Agent(subagent_type="dart-backend" | "dart-parser" | "dart-frontend",
      model="opus",
      description="<간단한 작업 설명>",
      prompt="<요청 원문 + 관련 배경 + Phase 1에서 파악한 순서상 이전 단계 산출물 요약>")
```

- `run_in_background: false`로 순차 실행한다 — 다음 단계(다른 영역 위임 또는 리뷰)가
  이 결과에 의존하므로 결과를 받아야 진행할 수 있다.
- 여러 영역을 순차 위임할 때는 이전 에이전트의 최종 보고(무엇을 바꿨는지)를 다음
  에이전트의 prompt에 그대로 포함시켜, 두 번째 에이전트가 처음부터 코드베이스를
  재탐색하지 않게 한다.

## Phase 3: 검증 위임 (생성 → 검증)

구현 내용에 따라 읽기 전용 리뷰 에이전트를 호출한다:

| 변경 영역 | 리뷰 에이전트 |
|---|---|
| 백엔드/파이프라인/파서 코드 변경 | `dart-qa` |
| 프론트엔드 코드 변경 | `dart-design-review` |
| 둘 다 변경 | 둘 다 순차 호출 (dart-qa 먼저 — 데이터 계약이 깨지면 프론트 검토가 무의미하므로) |

```
Agent(subagent_type="dart-qa" | "dart-design-review",
      model="opus",
      description="<리뷰 대상 요약>",
      prompt="<이번에 무엇을 바꿨는지 + 리뷰해야 할 파일/범위>")
```

리뷰 에이전트가 문제를 발견하면(빈 목록이 아니면), 해당 구현 에이전트에게 리뷰 결과를
그대로 전달해 재위임한다 — 오케스트레이터가 직접 코드를 고치지 않는다. 수정 후 같은
리뷰 에이전트로 재검증한다(최대 2회 반복 — 그 이상 반복되면 Phase 4에서 사용자에게
판단을 넘긴다).

## Phase 4: 결과 보고

사용자에게 다음을 요약한다: 어떤 에이전트가 무엇을 바꿨는지, 리뷰 결과(이상 없음 /
발견·수정 내역), 남은 이슈(있다면). 리뷰가 반복 실패했거나 설계 판단이 필요한 지점이
있으면 사용자에게 직접 질문한다 — 오케스트레이터가 임의로 결정하지 않는다.

작업이 CLAUDE.md에 기록할 만한 설계 변경(스파이크 결과, 아키텍처 변경, 확정된 정책
등)을 포함하면, 위임받은 에이전트가 이미 CLAUDE.md를 갱신했는지 확인한다 — 이
프로젝트의 확립된 관행(각 에이전트 정의의 "작업 시작 전 필수" 참고)이다.

## 에러 핸들링

에이전트 호출이 실패하거나(도구 오류, 타임아웃) 응답이 불확실하면 1회 재위임한다.
재실패하면 사용자에게 상황을 있는 그대로 보고하고 다음 행동을 확인받는다 — 코드베이스
작업이라 "누락을 명시하고 조용히 진행"하면 안 된다(생성형 콘텐츠 파이프라인과 달리
빠진 부분이 곧 버그가 된다).

## 테스트 시나리오

**정상 흐름**: "STEP 3 지역 필터에서 상장사가 안 걸러진다" → Phase 1에서 dart-backend로
분류 → dart-backend 위임·수정 완료 → Phase 3에서 dart-qa 호출 → "이상 없음" → Phase 4
보고.

**에러 흐름**: "재무제표에 부채비율 항목 추가해줘" → dart-parser(파서 필드 추가) →
dart-backend(DB 컬럼·API 반영, 이전 dart-parser 산출물 요약 전달) → dart-frontend(화면
노출) → dart-qa(백엔드/파서 리뷰) 호출 시 "resume 체크포인트가 새 필드를 놓친다" 발견 →
dart-backend에 재위임해 수정 → dart-qa 재검증 "이상 없음" → dart-design-review(프론트
리뷰) → 보고.
